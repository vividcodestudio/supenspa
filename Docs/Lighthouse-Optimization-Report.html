1. Preconnect Resource Hints Performance

What: Adding <link rel="preconnect"> tags for Google Fonts and Cloudflare CDN in the <head> of every page.

🇮🇳 Kya: Browser ko pehle se bata denge ki Google Fonts aur CDN se connection banao — taaki fonts jaldi load ho.

Why: Without preconnect, the browser discovers these external servers only after parsing CSS. This adds 200-500ms delay. Preconnect establishes the connection early.

🇮🇳 Kyun: Bina preconnect ke, browser fonts tab dhundhta hai jab CSS padh chuka hota hai — isse 200-500ms extra time lagta hai.

Code added:

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link rel="preconnect" href="https://cdnjs.cloudflare.com" crossorigin>

Risk: Zero ❌ — No visual change

Applied to: All 34+ HTML pages

2. Remove Duplicate @import in CSS Performance Best Practices

What: Removing the @import url('...fonts...') line from styles.css (line 7).

🇮🇳 Kya: styles.css ke andar se Google Fonts ka @import hatayenge.

Why: The same Google Fonts are already loaded via <link> in every HTML page. The CSS @import is a duplicate request that blocks rendering — the browser has to first download CSS, then discover the @import, then download fonts. This is the #1 reason for slow First Contentful Paint.

🇮🇳 Kyun: Yahi fonts already HTML mein <link> se load ho rahe hain. CSS ka @import ek duplicate render-blocking request hai — browser pehle CSS download karta hai, phir @import dhundhta hai, phir fonts — yeh chain bahut slow hai.

Before: @import url('https://fonts.googleapis.com/css2?family=...&display=swap');

After: /* Font loaded via HTML <link> with preconnect for better performance */

Risk: Zero ❌ — Fonts wahi rahenge, sirf doosra duplicate request hatega

3. Add width/height to All Images Performance

What: Adding width and height HTML attributes to all <img> tags.

🇮🇳 Kya: Sab images mein width aur height attributes add karenge.

Why: Without explicit dimensions, the browser doesn't know how much space an image needs until it finishes loading. This causes Cumulative Layout Shift (CLS) — the page "jumps" as images load. CLS is the #1 reason your Performance score is 73 instead of 95+. Adding width/height lets the browser reserve space beforehand.

🇮🇳 Kyun: Bina width/height ke browser ko nahi pata image kitni jagah legi — toh jab image load hoti hai toh page "jump" karta hai. Isko CLS kehte hain. Yeh sabse bada reason hai Performance score kam hone ka. Width/height dene se browser pehle se jagah reserve kar leta hai.

Important: CSS still controls the actual visual size. These attributes only help the browser calculate aspect ratio before loading.

🇮🇳 Important: Image ka dikhne wala size CSS se hi control hoga. Yeh attributes sirf browser ko aspect ratio batate hain.

Risk: Zero ❌ — No visual change, CSS overrides these values

Applied to: ~49 images in index.html + all other pages

4. Lazy-load Mega Menu Images Performance

🇮🇳 Kyun: Mega menu by default hidden hai. Par uske andar ki 16 images page open hote hi load hoti hain — ~2MB bandwidth waste. Lazy loading se yeh sirf tab load hongi jab user menu kholega.

Risk: Zero ❌ — Images will load on hover/click, invisible to user

5. Defer JavaScript Performance

What: Adding defer attribute to <script src="script.js">.

🇮🇳 Kya: script.js ko defer karenge.

Why: Without defer, the browser stops rendering the page while downloading and executing JavaScript (32KB). With defer, the script downloads in parallel and executes after the HTML is fully parsed — faster first paint.

🇮🇳 Kyun: Bina defer ke browser page render rokta hai jab tak JS download nahi ho jata (32KB). defer se JS parallel download hota hai aur HTML parse hone ke baad execute hota hai — page jaldi dikhta hai.

Before: <script src="assets/js/script.js"></script>

After: <script src="assets/js/script.js" defer></script>

Risk: Zero ❌ — DOMContentLoaded events still fire correctly with defer

7. Skip-to-Content Link + ARIA Roles Accessibility

What: Adding an invisible "Skip to main content" link + role="contentinfo" on footer + id="main-content" on first section.

8. Hamburger Menu: div → button Accessibility

🇮🇳 Kyun: Screen readers <div> ko button nahi samajhte. <button> banana se keyboard + screen reader dono se accessible hoga. CSS bilkul same rahega.

Risk: Zero ❌ — Same CSS applies to <button>, visual appearance unchanged
